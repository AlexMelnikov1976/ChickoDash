## 1. Что это и зачем

**Chicko Analytics** — аналитическая платформа для франчайзи сети ресторанов Chicko. Показывает ключевые метрики, сравнивает каждый ресторан с сетью, строит динамику, выдаёт рекомендации.

**Пользователи:** 42 франчайзи сети Chicko + управляющая компания.

**Текущий фокус:** MVP + защита IP завершены. Следующие задачи — UX-полировка, товарный знак, пилот.

## 5. Архитектурные решения

### 5.1–5.33 — предыдущие (см. v3.15)

### 5.34 Top-down UX — прогноз сверху, детали ниже (19.04.2026)

**Решение:** Первый экран дашборда перестраивается: сначала прогноз «куда идём», потом детали. Руководителю не нужно проваливаться в 6 KPI если всё в норме.

**Причина:** Цитата владельца: «Меня как руководителя в первую очередь интересует прогноз куда мы идём, а уже потом средняя выручка и средний чек. Если у меня всё хорошо — зачем мне проваливаться в детали?»

**Статус:** Идея зафиксирована, реализация — задача #71 в Фазе 1.4.

### 5.35 Like-for-like сравнения по дням недели (20.04.2026)

**Решение:** Все сравнения в дашборде используют **like-for-like** логику по дням недели. Средний чек за субботу сравнивается с нормой суббот (за 90 дней), а не с усреднённой неделей.

**Три уровня сравнения:**
1. **Ваша норма** (основная база) — медиана текущего ресторана за те же дни недели за 90 дней. Показывается если история ≥ 14 дней. Это то, чем владелец реально управляет.
2. **Медиана сети** (контекст) — медиана всех 43 ресторанов за те же дни недели за 90 дней. Fallback если «нормы» нет.
3. **Топ-25% сети** (потенциал) — 75-й перцентиль. Показывает куда можно расти.

**Причина:** Инсайт владельца: «Сравнивать средние чеки за субботу и за понедельник неверно». Это фундаментальная ошибка предыдущей версии.

**Реализация:** 2 SQL-запроса при загрузке (`loadDowProfiles`), функция `dowBenchmarks(ts)` агрегирует профили по DOW текущего периода.

**Что это дало:**
- «Низкий средний чек» (при чеке 1439₽ vs P75 1970₽) → «Средний чек в вашей норме» (vs своя норма 1448₽, разница -0.6%). Честно и не паникует.
- Доставка 26.5% при медиане сети 15.9% → не «хуже сети», но «хуже вашей нормы 32.3%» → актуальная проблема.

### 5.36 Виджет обратной связи — сквозная автоматизация (20.04.2026)

**Решение:** В дашборде добавлена кнопка 💬 (floating, правый нижний угол). Клик открывает модалку с 4 категориями (Баг / Идея / Данные неверны / Непонятно) + текстовое поле. Фидбек уходит:

```
Дашборд → POST /api/feedback (Worker, JWT auth)
  → waitUntil → n8n webhook (fire-and-forget)
    → Notion (база «Обратная связь», Kanban по статусам)
    → Telegram (уведомление в чат -1003396450964)
```

**Причина:** Франчайзи начинают пользоваться дашбордом. Нужен канал обратной связи без барьеров — прямо из интерфейса, без email/WhatsApp/устных просьб.

**Notion база:** Колонки: Тема, Категория, Статус (Новый/В работе/Готово/Отклонён), Пользователь, Ресторан, Приоритет (P0-P3), Получено. Kanban-вьюшка по статусам.

**Secrets:** `FEEDBACK_WEBHOOK` (n8n Production URL). Notion API key — в n8n credentials (HTTP Header Auth).

### 5.37 Прогноз выручки — Алгоритм Г с YoY-коррекцией (20.04.2026)

**Решение:** Блок прогноза на месяц — первый элемент на главной, выше KPI. Алгоритм Г (гибрид):

1. ≥7 дней данных в текущем месяце → DOW-медианы текущего месяца (самый точный)
2. <7 дней + есть прошлый год → DOW прошлого года × коэффициент YoY (k = выручка последнего полного месяца / тот же месяц год назад)
3. Fallback → 90-дневные DOW-профили (`MY_DOW`/`NET_DOW`)

**Сезонность:** Коэффициент YoY автоматически корректирует данные прошлого года (сезон март–август vs несезон сентябрь–февраль). К 8-му дню месяца переключается на реальные данные.

**Галочка «Вся сеть»:** Переключатель в хедере (`toggleNetworkView`). При включении R = `buildNetworkR()` (агрегат всех ресторанов: суммы выручек, взвешенные %, средний чек = выручка/чеки). Все блоки страницы (KPI, алерты, инсайты, прогноз) пересчитываются.

**Причина:** Цитата владельца: «Меня как руководителя в первую очередь интересует прогноз куда мы идём».

### 5.38 Защита IP — перенос логики с клиента на сервер (20.04.2026)

**Решение:** Вся бизнес-логика (SQL-запросы к mart-таблицам, расчёты прогноза, like-for-like агрегации) переезжает из `dashboard.ts` в Worker API. Клиент получает готовые агрегированные числа через JWT-защищённые endpoints.

**Причина:** `dashboard.ts` отдаётся в браузер целиком (240 КБ) — любой может открыть View Source и скопировать все формулы, SQL-запросы и алгоритмы. Полная защита невозможна, но цель — **сделать копирование дороже подписки**.

**План из 3 шагов:**
1. **Фаза 2.1** ✅ (20.04) — `loadDowProfiles` → `GET /api/dow-profiles?restaurant_id=N`. Скрыта схема mart_restaurant_daily_base для DOW-профилей.
2. **Фаза 2.2** ⏳ — `computeForecast` → `GET /api/forecast?restaurant_id=N`. Алгоритм Г с YoY-коррекцией уедет на сервер.
3. **Фаза 2.3** ⏳ — закрытие `/api/query` (универсальный SQL-прокси) → замена на whitelisted specific endpoints. После этого клиент физически не сможет выполнить произвольный SQL.

**Сопутствующие меры (не-технические):**
- Регистрация System360 как товарного знака через Роспатент (10-15 тыс ₽ через агентство)
- Оформление режима коммерческой тайны (приказ + перечень сведений + NDA в договоры)
- Шаблон договора с отчуждением прав для подрядчиков
- Главный моут — накопленные данные в ClickHouse за месяцы работы (невозможно скопировать из исходников)

### 5.39 AI-инсайты через Claude Code Routines + Notion-кеш (направление, не реализовано)

**Решение (архитектурное направление):** На следующем этапе после пилота — добавить в дашборд блок «🤖 AI-инсайт дня». Инсайты генерируются раз в сутки (в 08:00) через Claude Code Routine, пишутся в Notion, дашборд читает последнюю запись. Это **отложенное** архитектурное решение, фиксируется в паспорт чтобы не забыть путь.

**Причина:** Пилотные пользователи хотят не «таблицу цифр», а «объяснение что произошло и что делать». On-demand AI в момент открытия дашборда (клиент → Claude API → 15-30с ответа) имеет три проблемы: (1) каждый клиент = платный запрос, стоимость линейна по трафику; (2) плохой UX из-за задержки; (3) недетерминированные ответы — два пользователя на одни и те же цифры получат разные тексты. Batch-генерация раз в сутки решает все три: константная стоимость, мгновенный UX, один каноничный ответ.

**Архитектура:**

```
08:00 ежедневно
  → Claude Code Routine (на инфре Anthropic, компьютер не нужен)
    → POST n8n webhook
      → n8n: агрегаты из ClickHouse (вчера vs неделя назад)
      → возвращает JSON с метриками
    ← Routine интерпретирует, пишет 3-5 инсайтов в Notion
      (база «AI Insights», поля: дата, ресторан, текст, приоритет)

В любой момент
  Дашборд → GET /api/ai-insights?restaurant_id=N → Worker
    → Notion API (читает последнюю запись для данного ресторана)
    → возвращает render-ready JSON
  Клиент → отображает в блоке «Инсайт дня»
```

**Почему именно Claude Code Routines, а не n8n + Claude API напрямую:**
- Routines — нативная инфра Anthropic, research preview с апреля 2026. Не нужно содержать cron в n8n, не нужно управлять секретами Claude API в n8n.
- Запуск на Max-плане — 15 Routine/день включены в подписку (нам нужно 1-2 для Чико + Puls + запас).
- Уже есть MCP-коннекторы: Notion, n8n, Gmail, Google Drive. Routine подключается к ним напрямую.

**Почему Notion как кеш, а не своя таблица в ClickHouse:**
- Notion уже интегрирован (через виджет обратной связи, решение 5.36).
- Редактируемо людьми — если Routine сгенерирует кривой инсайт, владелец может поправить текст вручную до того как клиент увидит.
- Даёт бесплатную админку без разработки.

**Переиспользование:** Паттерн (Routine → n8n → CH → Notion → дашборд) переносится 1-к-1 на Puls (кинотеатры) и любую следующую вертикаль. Это ядро будущего SaaS-шаблона System360.

**Ограничения и риски, которые нужно держать в голове:**
- **Research preview** — Anthropic может менять/ломать API. Для пилота допустимо, для production через 3+ месяца — нужен plan Б (fallback на обычный cron в n8n + Claude API). Архитектура должна быть такой, чтобы смена триггера не ломала остальное — именно поэтому n8n стоит посредником, а не вызывается Routine'ом напрямую.
- **Stateless запуски** — Routine не помнит прошлых выполнений. Вся история — в Notion. Design-требование: каждый промпт должен тянуть контекст из Notion сам.
- **Лимиты триггеров** — только hourly/nightly/weekly, нет произвольных cron. Для утренних инсайтов — достаточно. Если позже понадобится «пересчитать после обновления данных» — триггерим через HTTP endpoint Routine из n8n после успешной загрузки в CH.
- **Качество инсайтов** — главный риск. Первая неделя после запуска — ручная ревизия каждого утреннего вывода, итерация промпта. До тех пор блок «Инсайт дня» в дашборде скрыт для пользователей.
- **Зависимость от ClickHouse MCP** — нативного коннектора нет, идём через n8n. Это плюс (n8n можно менять независимо) и минус (ещё одна точка отказа между Routine и данными).

**Оценка трудозатрат:** ~15-20 часов суммарно (не за одну неделю — реалистично 2-3 недели календарно с тестированием):
- Настройка Claude Code + первой Routine: 1-2ч
- n8n workflow для агрегатов CH → JSON: 3-4ч
- Промпт-инжиниринг (итерации до приемлемого качества): 4-6ч
- `/api/ai-insights` endpoint в Worker + Notion API интеграция: 2-3ч
- Блок «Инсайт дня» в `dashboard.ts`: 2-3ч
- Первая неделя ручной ревизии и калибровки: 3-5ч по часу в день

**Приоритет:** после модуля анализа блюд (Волна 6, раздел 12). Причина: без детализации до блюд AI-инсайты сведутся к общим выводам о выручке/среднем чеке и быстро выдохнутся. С блюдами можно давать конкретику: «блюдо X упало на -30% за 2 недели, у конкурентов в районе оно в среднем на 15% дешевле — проверьте цену».

## 12. План модуля «Анализ продаж по блюдам» (Волна 6)

**Цель:** дать франчайзи понимание какие блюда приносят деньги, какие — балласт, что продавать активнее, что убрать из меню.

**Объём:** 4 500 чеков/день × средняя глубина 6 позиций = 27 000 строк позиций в день по всей сети.

**Требуемые таблицы в ClickHouse:**

```
chicko.fact_check_items (raw — каждая позиция каждого чека)
  check_id, restaurant_id, dept_uuid, report_date,
  item_id, item_name, category, subcategory,
  qty, price_rub, cost_rub, discount_rub, revenue_net_rub
  ENGINE = MergeTree
  PARTITION BY toYYYYMM(report_date)
  ORDER BY (restaurant_id, report_date, item_id)

chicko.mart_menu_daily (агрегат по дням и блюдам — для быстрых запросов)
  report_date, restaurant_id, item_id, item_name, category,
  qty_sold, revenue_rub, foodcost_rub, margin_rub, checks_count
  ENGINE = SummingMergeTree
  PARTITION BY toYYYYMM(report_date)
  ORDER BY (restaurant_id, report_date, item_id)
```

**Пайплайн данных (n8n):**
- Источник: iiko OLAP транзакционный выгруз (items by check)
- Периодичность: ежедневно за предыдущий день
- Ретро-загрузка: с 2024-01-01 помесячно (чтобы не словить timeout iiko)

**Серверные эндпоинты (новые файлы `src/menu_*.ts`):**

| Эндпоинт | Возвращает | Payload |
|---|---|---|
| `GET /api/menu-top?restaurant_id=N&start=...&end=...&limit=20&sort=revenue\|qty` | Топ-N блюд по выручке или количеству | ~5 КБ |
| `GET /api/menu-abc?restaurant_id=N&start=...&end=...` | ABC-классификация всех позиций меню | ~20-50 КБ |
| `GET /api/menu-item-dynamics?restaurant_id=N&item_id=X&start=...&end=...` | Динамика одной позиции по дням | ~3-5 КБ |
| `GET /api/menu-tail?restaurant_id=N&threshold=<N продаж в неделю>` | «Хвост меню» — кандидаты на удаление | ~10 КБ |

Все эндпоинты — как и в Фазе 2: JWT-auth, whitelisted параметры, агрегаты на сервере, клиент получает render-ready JSON.

**Клиентская вкладка «Меню»:**
- Топ-20 таблица (выручка / количество / маржа, переключатель)
- Chart.js bar chart категорий
- ABC-пирамида (% блюд А = % выручки А)
- Выбор блюда → динамика продаж за период (отдельный fetch)
- Фильтр «хвост меню» — блюда с <N продаж/нед, с кнопкой экспорта для обсуждения

**Производительность:**
- CH-запросы по `mart_menu_daily`: 1-5 мс
- Worker response: 50-100 мс
- Клиент Chart.js: 200-500 блюд в таблице — ок с виртуальной прокруткой

**Что НЕ делаем:**
- Не отдаём сырые `fact_check_items` в браузер (payload 20+ МБ за месяц)
- Не делаем запросы без `restaurant_id` (защита от полной выгрузки через один запрос)
- Не храним `check_id` в клиенте (анонимизация)

**Оценка трудозатрат:**
- Таблицы CH + mart + n8n пайплайн: ~6-8 часов
- 4 серверных эндпоинта: ~4-5 часов
- Клиентская вкладка + графики: ~5-7 часов
- **Итого: 15-20 часов на полноценный модуль**

**Риски:**
- iiko OLAP может тормозить на транзакционных выгрузах → ретро помесячно
- Глубина чека может варьироваться (1-20 позиций) → кейсы для тестов
- Классификация товаров в iiko может дублироваться/меняться → нужна слабая нормализация по `item_name`, не только `item_id`

**Следующий шаг после пилота:** рекомендательный движок — «блюдо X упало на -30% за месяц → проверьте цену у конкурентов» или «блюдо Y в топ-5 по марже, но в топ-30 по продажам → продвигайте официантами».

#### Реализация

**`src/security.ts` (+96 строк):**
- `RateLimitConfig` — interface `{ limit, windowSec }`
- `RATE_LIMIT_DATA` = 60 запросов/мин/user для аналитических endpoints
- `RATE_LIMIT_FEEDBACK` = 10 запросов/мин/user для `/api/feedback`
- `rateLimitOrResponse(kv, key, config, request)` — возвращает Response(429) или null
- Fixed-window counters в KV (переиспользуем `MAGIC_LINKS` namespace с префиксом `rl:`)
- FAIL-OPEN при ошибках KV — легитимные пользователи не должны ловить 429 из-за инфраструктурного сбоя

- `/api/dow-profiles` (в `dow_profiles.ts`)
- `/api/forecast` (в `forecast.ts`)
- `/api/feedback` (в `index.ts`, с `RATE_LIMIT_FEEDBACK`)

**Паттерн 2-step в каждом handler'е:**
```ts
const payload = await validateToken(...);
if (!payload) return ...401;

const rl = await rateLimitOrResponse(env.MAGIC_LINKS, `data:${payload.user_id}`, RATE_LIMIT_DATA, request);
if (rl) return rl;
```

JWT проверка (дешёвая) идёт первой, rate limit — второй. Неавторизованные запросы отсекаются до KV-операций. Это правильно для защиты легитимных пользователей, но означает что защиты от DoS на неавторизованные endpoint'ы нет — добавим при необходимости отдельным слоем.

### Урок Phase 2.4a — четыре отката из-за плохой сигнатуры

Сигнатура `jsonResponse(body, status?, request?)` с двумя опциональными аргументами оказалась ловушкой:

```ts
// Я писал в коде:
jsonResponse({ ok: true }, request)  // ← хотел: body + request
// JavaScript интерпретировал:
jsonResponse({ ok: true }, status=request)  // ← request попал в status!
// Результат: new Response(..., { status: <Request object> })
// → "Responses may only be constructed with status codes in the range 200-599"
```

Усугубилось:
1. Я применил Python-скрипт автозамены ко всем jsonResponse вызовам в 4 файлах
2. Скрипт работал, но инвертировал порядок: `(body, status, request)` ↔ `(body, request, status)` в разных проходах
3. Маскировка ошибок (по тому же аудиту) скрыла причину собственного бага — клиент получал генерический «Request failed» вместо traceback'а

**Итог:** 4 деплоя с откатом, потерян час. Прод восстановлен через `git revert` каждый раз.

**Решение, принятое в Phase 2.4a:** сигнатура `jsonResponse(body, request, status?)` — `request` обязателен (CORS должен работать всегда), `status` опционален (default 200). Type system теперь не позволит забыть.

**Уроки на будущее (зафиксированы):**
1. **Никогда** не делать TS-функцию с двумя опциональными аргументами одного семантического контекста
2. После автозамены через скрипт **всегда** визуально проверять 5-10 случайных вызовов перед `git push`
3. При маскировке ошибок (требование security) **всегда** оставлять полный traceback в `console.error` для `wrangler tail`
4. При первом подозрительном symptom — **сразу** запускать `wrangler tail` во втором терминале, не угадывать причину

### Полная инструкция по аудиту

## 15. Коммерческая стратегия (гипотезы до пилота)

> **Статус:** гипотезы для внутреннего обсуждения и переговоров с пилотными клиентами. Цифры не проверены реальной продажей, калибруются по результатам Волны 5 (пилот с 3-5 франчайзи). После пилота — обновление раздела с пометкой «факт вместо гипотезы».
>
> **Источник:** внешний стратегический разбор (ChatGPT Pro, 21.04.2026) + фильтр Claude под реалии РФ-рынка.

### 15.1 Позиционирование

Chicko Analytics — **vertical intelligence platform для multi-unit restaurant сетей**, а не BI-дашборд. Ключевое отличие от дженерик-аналитики: система дневного управления, превращающая данные в операционные действия (утренние инсайты → задачи управляющему → отслеживание закрытия отклонений).

Лестница зрелости продукта:

1. **Intelligence layer (текущий этап):** метрики, прогноз, бенчмарки, like-for-like, аномалии. MVP закрыт к 20.04.2026.
2. **Action layer (Волна 7, см. 5.39):** AI-инсайты через Claude Routines, задачи управляющему, эскалация в Telegram, attribution эффекта.
3. **Control layer (2-3 года):** labor optimization, menu engineering, promo/price experiments, procurement signals. Конкуренция уже не с BI, а с operational platforms.

### 15.2 Таргет-сегмент (ICP)

| Критерий | Целевой профиль | Почему |
|---|---|---|
| Размер сети | 10-150 точек | Excel уже болит, all-in-one ERP ещё дорог |
| Форматы | Multi-format (fast casual, delivery, dine-in) | Выше ценность cross-source аналитики |
| Модель | Франчайзинг или холдинг с регионами | Нужна прозрачность между точками |
| Стек POS | iiko (обязательно) + 1С (желательно) | Технический вход уже решён |
| Роль покупателя | CEO/owner + CFO + COO | Решение покупают несколько лиц совместно |

**Первые клиенты** — через ChicoGroup и её сеть знакомств. Пилот: 3-5 франчайзи в разных городах + разных форматах (для проверки переносимости промптов/моделей).

### 15.3 Ценовая модель (РФ, гипотезы)

Базовая линейка — per-location/month, минимум 5 точек в сети:

| Тариф | Целевая сеть | Цена/точка/мес | Мин. чек/мес |
|---|---|---|---|
| **Starter** | 5-20 точек | 3-5 тыс ₽ | 20-50 тыс ₽ |
| **Pro** | 20-100 точек | 5-10 тыс ₽ | 100-500 тыс ₽ |
| **Enterprise** | 100+ точек, франчайзинг | custom, целевой ACV 500 тыс - 2 млн ₽/год | — |

**Важно:** эти цифры — мой (Claude) фильтр на западные ориентиры ($79-600/location/month у R365, MarginEdge, 7shifts) через WTP российского рынка. В РФ аналитика для ресторанов продаётся **в 3-5× дешевле** чем в США/ЕС — iiko.Office, Quick Resto, 1С-Рарус как якоря. Если пилот покажет что рынок готов платить больше — цифры пересматриваем вверх.

**Дополнительная монетизация:**

- **Onboarding / внедрение:** 50-200 тыс ₽ единоразово (настройка коннекторов, обучение управляющих, кастомизация KPI-модели под сеть). Почти обязательно для enterprise-сегмента.
- **Add-on модули** (после MVP базовой линейки):
  - Forecasting Pro (продвинутое прогнозирование с seasonality/events)
  - Food Cost / Theoretical Variance (требует интеграции с техкартами iiko)
  - Delivery & Channel Intelligence (каналовая экономика + API агрегаторов, см. Волна 6)
  - Executive Board Pack (отчёты для CFO/акционеров)
  - AI Recommendations / Action Layer (Волна 7, см. 5.39)
- **Сервисная выручка:** кастомные коннекторы, разовые отчёты, консалтинг по KPI-модели. На раннем этапе может быть 20-40% общей выручки.

- Потолок 100-200 млн ₽ ARR — без международного выхода. Международный рынок — другой проект с другой командой, сейчас не закладываем.

### 15.5 Конкурентный ландшафт (РФ)

**Прямые конкуренты (угроза):**

- **iiko.Office** — встроенная аналитика, бесплатная часть при подписке на iiko POS. Покрывает базовые KPI, плоха в кросс-ресторанных сравнениях и прогнозировании. Главный риск — iiko может встроить более сильную аналитику в core-продукт.
- **Quick Resto, 1С-Рарус, Syrve, Poster** — аналогичные POS с встроенной аналитикой. Слабее iiko по распространению.

**Косвенные конкуренты:**

- **Excel + Google Sheets** — главный конкурент в сетях до 10 точек. Бесплатно, гибко, но не масштабируется.
- **Power BI / Tableau + свой разработчик** — у крупных сетей (100+ точек) есть in-house аналитики. Против них идти бесполезно.
- **Внешние агентства на аутсорсе** — «привозят» дашборды за 200-500 тыс ₽/разовый проект. Плохо подходят для daily operations.

**Западные аналоги (для словаря и направления, не для прямой конкуренции):**
- Restaurant365 ($350+/location/month, all-in-one back office)
- Tenzo (performance + AI recommendations)
- MarginEdge ($350/location/month, food cost + daily P&L)
- Deliverect Pulse (delivery/marketplace analytics)
- 7shifts (labor scheduling + ops)

В РФ этих продуктов нет. Полезно знать для понимания куда движется рынок и как правильно формулировать ценность, но сравнение «мы как Restaurant365» на переговорах с РФ-клиентом работать не будет — они о нём не слышали.

### 15.6 Moat — чем защищаемся

Код сам по себе moat не создаёт — это зафиксировано в 5.38. Реальная защита:

1. **Глубина интеграции с iiko:** коннекторы, обработка специфики (акции, модификаторы, отменённые чеки, техсбои). Накапливается эмпирически, не воспроизводится с нуля.
2. **Накопленные peer benchmarks:** чем больше сетей в системе, тем ценнее сравнительная аналитика. Сетевой эффект для данных.
3. **Встроенность в операционные процессы:** daily digests, action tracking, эскалация в Telegram превращают Chicko в обязательный ежедневный инструмент. Высокие switching costs.
4. **Кастомные модели под сеть:** промпты AI-инсайтов, KPI-профили, формулы скоров — подгоняются под ICP конкретной сети. Переключение = потеря калибровки.
5. **Коммерческая тайна** (режим + договоры + NDA): формулы, пороги, playbook'и.
6. **Товарный знак System360** (в процессе): базовая защита от клонов.

### 15.7 Риски масштабирования

- **Продуктовый:** если останется «дашбордом» без action layer — восприятие как nice-to-have, низкий WTP. Защита: ускорить Волну 7 (AI-инсайты) после пилота.
- **Конкурентный:** iiko может добавить аналогичную аналитику в core. Защита: глубина иниции в процессы клиента + кастомные модели.
- **Интеграционный:** с ростом сети качество данных из iiko/1С становится дороже чем UI. Защита: инвестировать в коннекторный слой как платформенный компонент.
- **Доверительный/юридический:** action-рекомендации = ответственность. Защита: audit trails, disclaimers, явное «рекомендация — решение принимает человек» в UI.

### 15.8 Что проверяем в пилоте (чтобы откалибровать этот раздел)

Пилот с 3-5 франчайзи должен ответить на:

1. **Готовы ли платить?** Если да — сколько? Если нет — почему (цена / ценность / страх смены).
2. **Какие модули использует реально?** Прогноз / бенчмарки / like-for-like / алерты. Что-то из этого может оказаться игнорируемым балластом.
3. **Частота открытия дашборда.** Цель — ежедневно. Если раз в неделю — продукт не вошёл в операционную рутину.
4. **Кто именно пользователь?** Владелец / управляющий / региональный менеджер. Влияет на roadmap фич для разных ролей.
5. **Что просят добавить первым?** Даст приоритет модулей из 15.3 (Food Cost / Delivery / Labor / Menu).
6. **На сколько болит Excel/iiko.Office прямо сейчас?** Калибровка ценового окна.

Результаты пилота → обновление раздела 15 с пометкой «факт вместо гипотезы» и пересмотром roadmap (Волны 6-7).

### 15.9 Правовая защита — ссылка

Детали в разделах 5.38 (архитектурное решение защиты IP) и 13 (security-аудит). Короткая сводка:

- **Первичный инструмент:** режим коммерческой тайны + контрактный контур (NDA в договоры, IP assignment для подрядчиков, anti-reverse-engineering clauses). Делаем сейчас, перед пилотом.
- **Товарный знак System360** — регистрация через агентство. ~10-15 тыс ₽, срок 6-8 месяцев.
- **Авторское право** — автоматическое, оформляем права компании на код/дизайн/документацию.
- **Патент** — **не первичный инструмент**. Patentability для чисто аналитических/AI/business-method решений в ЕС и РФ сильно ограничена (требование technical character). Имеет смысл только на узкие технические объекты (например: privacy-preserving multi-tenant benchmarking, специфическая схема edge+analytical backend). До этого уровня ещё не дошли.
