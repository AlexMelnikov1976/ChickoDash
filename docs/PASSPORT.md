## 1. Что это и зачем

**Chicko Analytics** — аналитическая платформа для франчайзи сети ресторанов Chicko. Показывает ключевые метрики, сравнивает каждый ресторан с сетью и топ-10, строит динамику, выдаёт рекомендации.

**Пользователи:** 42 франчайзи сети Chicko + управляющая компания.

**Текущий фокус:** **Фаза 1 полировки** по результатам AI-критики — 22 пункта P0 из backlog.

### 5.32 Модельный ФОТ не показываем (19.04.2026 вечер)

**Решение:** Подтверждаем [5.26](#526) окончательно. Поле `labor_cost_model_rub` из `mart_restaurant_daily_base` в UI **не используется**.

**Что проверено:** SQL-запрос по всем 43 ресторанам за последний день показал, что `labor_cost_model_rub / revenue_total_rub` у всех точек лежит в коридоре **22.9 — 24.3%**. Это **линейная функция от выручки** (норматив × выручка), не факт. Такой "ФОТ" не несёт информации, которой ещё нет в метрике выручки.

**Что ещё проверено:** флаг `has_actual_1c_data = false` у **всех 43 ресторанов**, включая Калининград. Значит реальной 1С-интеграции пока нет ни у кого.

**Следствие для `ebitda_margin_pct`:** тоже не показываем — она тоже линейная производная от модельного ФОТ, то есть от выручки.

**Что смотрим в будущее:** когда ты (Калининград) подключишь 1С и `has_actual_1c_data` станет `true` — можно показать **только тебе** дополнительный блок "Реальный P&L". Другие увидят стандартные 6 KPI + примечание. Это станет **маркетинговым стимулом** для остальных франчайзи подключить свою 1С. См. план как решение 5.34 (будущее).

### 5.33 Динамические перцентили из raw-таблицы, не из mart_benchmarks_daily (19.04.2026 вечер)

**Решение:** Сетевые бенчмарки (`NET` и `TOP10` в JavaScript) считаются **не из `mart_benchmarks_daily`**, а из `mart_restaurant_daily_base` **на лету** за тот же период, что выбрал пользователь.

**Причина:** `mart_benchmarks_daily` содержит **снимок за одно окно** (последние 30 дней перед `dt`). Если пользователь выбрал январь-апрель — его точка считается за 3.5 месяца, а сеть — за последний месяц. **Это главная претензия всех 4 AI-моделей**: сравнение несинхронизированных периодов создаёт иллюзию успеха/провала.

**Реализация:** Новая JS-функция `loadNetworkBenchmarks(start, end)` делает запрос:

```sql
SELECT
  quantile(0.50)(revenue_total_rub)      AS rev_median,
  quantile(0.90)(revenue_total_rub)      AS rev_p90,
  quantile(0.50)(avg_check_total_rub)    AS chk_median,
  ...
  count(DISTINCT dept_uuid)              AS rest_count
FROM chicko.mart_restaurant_daily_base
WHERE report_date BETWEEN '{start}' AND '{end}'
  AND is_anomaly_day = 0
  AND revenue_total_rub > 0
```

Вызывается при первой загрузке дашборда и при каждом изменении глобального календаря.

**Fallback при < 3 ресторанов:** `NET` и `TOP10` остаются нулями, UI показывает прочерки. Защита от некорректных статистик на малой выборке.

**Совместимость:** имена `NET.revenue`, `TOP10.revenue` и т.д. сохранены — весь остальной код (40+ использований) работает без правок.

**Таймаут/производительность:** ClickHouse за ~5000 строк (43 ресторана × 90 дней) отдаёт за десятки миллисекунд. Кеш на фронте **не делаем** — запрос достаточно быстрый, добавление кеша увеличит сложность без видимого выигрыша.

**Использованы медианы вместо средних** (`quantile(0.50)` вместо `avg()`). Причина: в ресторанной сети всегда есть 1-2 гиганта (ТЦ/центр города), которые смещают среднее. Медиана показывает **типичный ресторан**.
